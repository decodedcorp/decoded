# 기능 명세: 스크롤 애니메이션 & 레이지 로딩 시스템

**기능 브랜치**: `001-scroll-animation`
**생성일**: 2025-11-20
**상태**: 초안
**입력**: 사용자 설명: "opacity/transform 전환과 함께 IntersectionObserver 사용; data-src를 통한 이미지 레이지 로딩"

## 사용자 시나리오 & 테스트 _(필수)_

### 사용자 스토리 1 - 스크롤 시 부드러운 콘텐츠 등장 (우선순위: P1)

페이지를 스크롤하는 사용자로서, 뷰포트에 들어오는 콘텐츠 요소(카드, 이미지, 섹션)가 부드럽게 나타나기를 원합니다. 이를 통해 경험이 거칠지 않고 세련되고 전문적으로 느껴집니다.

**이 우선순위인 이유**: 핵심 시각적 경험 - 사용자가 즉시 인지하며 전체 애플리케이션의 품질 인식에 영향을 미칩니다. 이것 없이는 기능이 가치가 없습니다.

**독립 테스트**: 여러 카드 요소가 있는 페이지를 스크롤하고 각 카드가 320-420ms에 걸쳐 보이지 않는 상태(opacity: 0, translateY: 12px, scale: 0.98)에서 보이는 상태(opacity: 1, translateY: 0, scale: 1)로 부드럽게 전환되는지 확인하여 완전히 테스트할 수 있습니다. 즉각적인 시각적 세련미를 제공합니다.

**수락 시나리오**:

1. **Given** 화면 아래에 여러 카드가 있는 페이지, **When** 사용자가 아래로 스크롤하여 카드가 뷰포트에 진입, **Then** 카드가 320-420ms에 걸쳐 보이지 않음(opacity: 0, translateY: 12px, scale: 0.98)에서 보임(opacity: 1, translateY: 0, scale: 1)으로 전환
2. **Given** 연속된 여러 카드, **When** 카드가 뷰포트에 진입, **Then** 각 카드가 시차 지연으로 캐스케이드 효과를 만들며 나타남
3. **Given** 보이는 카드, **When** 사용자가 스크롤하여 카드가 뷰포트를 벗어남, **Then** 카드가 부드럽게 페이드 아웃(is-hidden 상태로 전환)

---

### 사용자 스토리 2 - 점진적 이미지 로딩 (우선순위: P2)

이미지가 있는 페이지를 보는 사용자로서, 이미지가 볼 때만 로드되기를 원합니다. 이를 통해 페이지가 더 빨리 로드되고 데이터 사용량이 줄어듭니다.

**이 우선순위인 이유**: 초기 페이지 로드 시간을 개선하고 대역폭 사용을 줄이는 성능 향상. P1이 작동하지 않아도 리소스 최적화를 추가하여 P1을 기반으로 합니다.

**독립 테스트**: 이미지가 있는 페이지를 로드하고 네트워크 요청을 모니터링하여 화면 아래 이미지가 스크롤할 때까지 로드되지 않는지 확인하여 테스트할 수 있습니다. 측정 가능한 성능 향상을 제공합니다.

**수락 시나리오**:

1. **Given** 화면 아래에 이미지가 있는 페이지, **When** 페이지 로드, **Then** 화면 위 이미지만 로드됨
2. **Given** data-src 속성이 있는 이미지가 뷰포트에 접근, **When** 이미지 컨테이너가 관찰 임계값에 진입, **Then** 브라우저가 data-src에서 이미지를 로드하고 표시
3. **Given** 이미 로드된 이미지, **When** 이미지가 나갔다가 뷰포트에 재진입, **Then** 이미지가 다시 로드되지 않음(loaded로 표시됨)

---

### 사용자 스토리 3 - 성능 최적화된 스크롤링 (우선순위: P3)

콘텐츠가 많은 페이지를 스크롤하는 사용자로서, 저사양 기기에서도 스크롤 중에 페이지가 부드럽고 반응성 있게 유지되기를 원합니다. 지연이나 버벅임이 없어야 합니다.

**이 우선순위인 이유**: 애니메이션이 사용자 경험을 저하시키지 않도록 보장하는 품질 향상. P1과 P2가 먼저 존재해야 하며, 그 다음 성능을 최적화합니다.

**독립 테스트**: 다양한 기기에서 스크롤 중 프레임 속도를 측정하고, Core Web Vitals(CLS ≤ 0.1, LCP ≤ 2.5s)를 확인하고, GPU 가속 렌더링을 확인하여 테스트할 수 있습니다. 성능 규정 준수를 제공합니다.

**수락 시나리오**:

1. **Given** 애니메이션 카드가 있는 페이지, **When** 사용자가 빠르게 스크롤, **Then** 애니메이션이 레이아웃 스래싱 없이 60fps 유지
2. **Given** 치수가 있는 이미지, **When** 이미지 로드, **Then** 레이아웃 이동 없음(CLS ≤ 0.1)
3. **Given** 모바일 기기의 페이지, **When** 사용자가 애니메이션 콘텐츠를 스크롤, **Then** 인지할 수 있는 버벅임 없이 스크롤이 부드럽게 유지

---

### 엣지 케이스

- JavaScript가 비활성화되면 어떻게 되나요? (이미지는 네이티브 `loading="lazy"`를 폴백으로 사용해야 함)
- 시스템이 매우 빠른 스크롤을 어떻게 처리하나요? (IntersectionObserver는 성능 저하 없이 빠른 스크롤 이벤트를 효율적으로 처리해야 함)
- 이미지 로드에 실패하면 어떻게 되나요? (시스템은 애니메이션 흐름을 깨지 않고 깨진 이미지 상태를 우아하게 처리해야 함)
- 화면 위 이미지는 어떻게 처리되나요? (LCP 최적화를 위해 레이지 로딩 없이 즉시 로드해야 함)
- 사용자가 다시 위로 스크롤하면 어떻게 되나요? (이전에 로드된 이미지는 로드된 상태로 유지, 애니메이션은 is-visible/is-hidden 상태에 따라 다시 트리거될 수 있음)
- 시스템이 뷰포트 크기 조정을 어떻게 처리하나요? (IntersectionObserver는 새 뷰포트 치수에 따라 가시성을 자동으로 재계산해야 함)

## 요구사항 _(필수)_

### 기능 요구사항

- **FR-001**: 시스템은 15% 임계값의 IntersectionObserver를 사용하여 뷰포트에 진입/이탈하는 요소를 관찰해야 함
- **FR-002**: 시스템은 관찰된 요소에 초기 숨김 상태(opacity: 0, transform: translateY(12px) scale(0.98))를 적용해야 함
- **FR-003**: 시스템은 요소가 뷰포트에 진입할 때 `is-visible` 클래스를 추가하고 `is-hidden` 클래스를 제거해야 함
- **FR-004**: 시스템은 요소가 뷰포트를 이탈할 때 `is-hidden` 클래스를 추가하고 `is-visible` 클래스를 제거해야 함
- **FR-005**: 시스템은 data-delay 속성을 CSS 사용자 정의 속성으로 변환하여 시차 애니메이션 지연을 구현해야 함
- **FR-006**: 시스템은 이미지 컨테이너가 뷰포트에 진입할 때 data-src를 src로 교체하여 이미지를 레이지 로드해야 함
- **FR-007**: 시스템은 재진입 시 재로드를 방지하기 위해 이미지를 loaded(data-loaded="true")로 표시해야 함
- **FR-008**: 시스템은 점진적 향상으로 이미지에 네이티브 `loading="lazy"` 속성을 사용해야 함
- **FR-009**: 시스템은 GPU 가속 속성(opacity 및 transform)만 사용하여 애니메이션해야 함
- **FR-010**: 시스템은 애니메이션 카드 요소에만 `will-change: opacity, transform` 힌트를 적용해야 함
- **FR-011**: 시스템은 320-420ms 지속 시간과 함께 cubic-bezier(0.22, 1, 0.36, 1) 타이밍 커브를 사용해야 함
- **FR-012**: 시스템은 최적화된 트리거를 위해 rootMargin "0px 0px -10% 0px"으로 IntersectionObserver를 구성해야 함
- **FR-013**: 시스템은 관찰 콜백에서 O(1) 복잡도를 유지해야 함(클래스 토글만)
- **FR-014**: 이미지는 레이아웃 이동을 방지하기 위해 고정 치수(width/height 또는 aspect-ratio)를 지정해야 함
- **FR-015**: 히어로/화면 위 이미지는 LCP 최적화를 위해 미리 로드하거나 우선순위 힌트를 사용해야 함

### 주요 엔티티

- **관찰 가능한 요소**: 스크롤 애니메이션이 필요한 콘텐츠 요소(카드, 섹션), `.js-observe` 클래스와 시차 타이밍을 위한 선택적 `data-delay` 속성으로 표시
- **레이지 이미지**: 실제 이미지 URL을 포함하는 `data-src` 속성이 있는 이미지 요소, `src`는 초기에 설정되지 않거나 플레이스홀더, 네이티브 지원을 위한 `loading="lazy"`
- **애니메이션 상태**: CSS 클래스(`is-visible`, `is-hidden`)와 인라인 CSS 사용자 정의 속성(`--stagger`)으로 표현되는 요소의 시각적 상태

## 성공 기준 _(필수)_

### 측정 가능한 결과

- **SC-001**: 사용자가 콘텐츠를 스크롤할 때 갑작스러운 팝인 없이 부드럽고 전문적인 애니메이션을 인지
- **SC-002**: 10개 이상의 이미지가 있는 페이지에서 화면 아래 이미지 로드를 지연시켜 초기 페이지 로드 시간이 30-50% 개선
- **SC-003**: 중급 기기에서 애니메이션 전환 중 스크롤 성능이 60fps(프레임당 16.67ms) 유지
- **SC-004**: 페이지 수명 주기 전반에 걸쳐 누적 레이아웃 이동(CLS)이 ≤ 0.1 유지
- **SC-005**: 히어로/우선순위 콘텐츠에 대해 최대 콘텐츠풀 페인트(LCP)가 ≤ 2.5초
- **SC-006**: 320-420ms 지속 시간과 연속 카드에서 보이는 캐스케이드 효과로 애니메이션 타이밍이 자연스럽게 느껴짐
- **SC-007**: 보이는 이미지만 로드하여 초기 페이지 로드 시 네트워크 사용량이 40-60% 감소
- **SC-008**: 사용자가 최신 브라우저(Chrome, Firefox, Safari, Edge)에서 일관된 애니메이션 동작을 경험

## 가정

1. **브라우저 지원**: IntersectionObserver 지원이 있는 최신 브라우저를 대상으로 합니다(Chrome 51+, Firefox 55+, Safari 12.1+, Edge 15+). 이전 브라우저의 폴백은 애니메이션 없이 네이티브 `loading="lazy"`를 사용합니다.

2. **콘텐츠 구조**: 이 기능을 사용하는 페이지는 개별적으로 관찰하고 애니메이션할 수 있는 카드와 같은 컴포넌트로 콘텐츠가 구성됩니다.

3. **성능 목표**: 60fps 목표를 위한 최소 성능 기준으로 중급 모바일 기기(예: iPhone 12, Samsung Galaxy S21)를 가정합니다.

4. **이미지 형식**: 웹 전달에 적합한 합리적인 파일 크기의 표준 웹 이미지 형식(JPEG, PNG, WebP, AVIF)을 가정합니다.

5. **초기 상태**: 관찰 대상으로 표시된 요소는 페이지 로드 시 DOM에 렌더링됩니다(로드 후 동적으로 주입되지 않음, 필요한 경우 시스템이 늦은 추가를 처리해야 함).

6. **시차 타이밍**: 자연스러운 캐스케이드 효과에 적합한 요소당 40-80ms의 기본 시차 지연 증분을 가정합니다.

7. **뷰포트 임계값**: 15% 임계값과 -10% 하단 rootMargin이 과도한 재트리거를 피하면서 완전한 가시성 전에 애니메이션을 트리거하는 데 최적이라고 가정합니다.

8. **GPU 가용성**: 기기가 하드웨어 가속 CSS 변환 및 불투명도를 지원한다고 가정합니다(모든 최신 기기의 표준).

## 비목표

이 기능은 명시적으로 다음을 포함하지 않습니다:

- opacity/transform 외의 사용자 정의 애니메이션 유형(예: 색상 변경, width/height 애니메이션)
- 애니메이션 라이브러리 또는 프레임워크(순수 CSS/JS 구현)
- 단순 시차 외의 복잡한 애니메이션 시퀀싱 또는 안무
- 패럴랙스 효과 또는 스크롤 연동 애니메이션
- 비디오 레이지 로딩(이미지에만 집중)
- 점진적 이미지 로딩 기술(blur-up, LQIP)
- 모션 감소 선호 사용자를 위한 폴백 UI(prefers-reduced-motion을 통해 별도로 처리해야 함)
- 애니메이션에 대한 서버 측 렌더링 고려
- 터치 제스처 기반 애니메이션
- 시각적 애니메이션 외의 스크롤 트리거 JavaScript 상태 변경
