# Pitfalls Research

**Domain:** Explore/Editorial data integration — existing Next.js 16 app with Orval-generated hooks, GSAP FLIP transitions, intercepting routes, and a Rust/Axum backend
**Researched:** 2026-04-01
**Confidence:** HIGH (all findings derived from direct codebase inspection, not training assumptions)

## Critical Pitfalls

### Pitfall 1: `has_magazine` exists in Rust DTO but is absent from `openapi.json`

**What goes wrong:**
The Rust `PostListQuery` struct (`packages/api-server/src/domains/posts/dto.rs`, line 180) defines a `has_magazine: Option<bool>` parameter. However, the committed `openapi.json` does NOT include this parameter under `GET /api/v1/posts`. The generated `ListPostsParams` type (`lib/api/generated/models/listPostsParams.ts`) therefore has no `has_magazine` field. The current `useInfinitePosts` hook in `useImages.ts` falls back to a **Supabase direct query** with a code comment `// Note: hasMagazine filter not yet supported via Supabase query`, silently ignoring the `hasMagazine` prop.

**Why it happens:**
The backend parameter was added to the Rust DTO and service layer but was not propagated to the OpenAPI spec. Since `openapi.json` is generated by utoipa on the Rust side and manually committed, the spec drifts from the implementation.

**How to avoid:**
Before writing any frontend filtering logic, regenerate `openapi.json` from the running backend (`cargo run` produces updated spec via utoipa) or manually add `has_magazine` to the spec's query parameters section using the established `python3` editing pattern. Then run `bun run generate:api` to regenerate `ListPostsParams`. Do not implement the filter by patching the Supabase query as a shortcut — the REST API path is the correct one.

**Warning signs:**
- `ExploreClient` with `hasMagazine={true}` returns the same posts as without the prop
- `ListPostsParams` has no `has_magazine` field after codegen
- `useInfinitePosts` contains the comment "hasMagazine filter not yet supported via Supabase query"

**Phase to address:**
Editorial data validation phase — this is the first integration dependency. Nothing else works correctly until this is resolved.

### Pitfall 2: Orval generates `useQuery` (single-page), not `useInfiniteQuery`

**What goes wrong:**
Orval 8.5.3 generates `useListPosts` as a standard `useQuery` wrapping `listPosts`. The generated hook fetches a single `PaginatedResponsePostListItem` — one page, not a stream of pages. The existing `useInfinitePosts` in `useImages.ts` works around this by bypassing Orval and doing a Supabase direct query with manual pagination. If someone tries to swap in the Orval-generated `useListPosts` hook for the explore grid, infinite scroll will break — the hook has no `fetchNextPage`, `hasNextPage`, or `getNextPageParam`.

**Why it happens:**
Orval generates infinite query hooks only when the OpenAPI spec includes a `x-orval-infinite` extension or when the config explicitly specifies cursor/page param mapping. Neither is configured in `orval.config.ts` for this project.

**How to avoid:**
Keep using the hand-written `useInfinitePosts` wrapper. If you need to migrate to the REST API endpoint (instead of Supabase direct), write a custom `useInfiniteListPosts` hook that calls the `listPosts` generated fetch function directly inside `useInfiniteQuery`, mapping `page` param from `pageParam`. Do not try to derive infinite scroll behavior from the generated `useListPosts` hook.

**Warning signs:**
- Someone imports `useListPosts` from generated code into ExploreClient
- Grid loads only the first page with no infinite scroll trigger
- `fetchNextPage` is undefined on the hook result

**Phase to address:**
Explore data connection phase — establish the correct hook pattern before wiring the grid.

### Pitfall 3: FLIP state stored in `transitionStore` is stale if card element is recycled by ThiingsGrid

**What goes wrong:**
`ExploreCardCell` captures GSAP `Flip.getState(target)` on click and stores it in `transitionStore`. ThiingsGrid is a class component with a physics engine that recycles DOM cell positions as the user scrolls (virtual infinite canvas, `MAX_RENDER_CELLS = 300`). If the user scrolls the grid after the FLIP state is captured but before the modal animation runs, the stored `originRect` DOMRect may no longer reflect the actual on-screen position of the card element (the element may have been repositioned or its cell reused).

**Why it happens:**
The FLIP state is captured at click time, but the animation runs asynchronously after Next.js navigation. In a virtual-scroll physics grid, the source element's position is dynamic.

**How to avoid:**
The current implementation already falls back gracefully: `ImageDetailModal` uses `imgSrc` from the store for immediate visual continuity even if the FLIP animation fails. Ensure the fallback path (storing `originRect` without Flip state) is tested. Do not attempt to re-query the source element by `data-flip-id` inside the modal — the element may no longer be in the DOM or at the right position.

**Warning signs:**
- FLIP animation plays from wrong screen position (card appears to fly from a different card)
- Console errors from `Flip.getState` on unmounted elements
- Animation visually correct on first click, incorrect after scrolling

**Phase to address:**
Explore detail page connection phase — test FLIP animation after scrolling at least 2 viewport heights before clicking.

### Pitfall 4: Intercepting route `@modal/(.)posts/[id]` only intercepts from same-depth routes

**What goes wrong:**
Next.js parallel/intercepting routes use segment depth matching. The interceptor `@modal/(.)posts/[id]` uses single-dot syntax `(.)`, which intercepts navigation from a sibling segment (one level up relative to the layout). Navigation to `/posts/[id]` from `/explore` renders the modal overlay correctly. Navigation from `/editorial` should also work because both `/explore` and `/editorial` are direct children of the root layout that contains the `@modal` slot.

What breaks: if someone adds a nested route (e.g., `/editorial/[magazineId]`) and links to `/posts/[id]` from within it, the single-dot `(.)` interceptor will NOT match — it would need `(..)posts/[id]` to go up two levels. Editorial detail pages or magazine-specific contexts at nested depths will bypass the modal and do a full-page navigation instead.

**Why it happens:**
The intercepting route scope is statically defined by the folder notation — it does not dynamically adjust for navigation depth.

**How to avoid:**
Keep all editorial/explore content at the root depth (i.e., `/editorial` and `/explore` as direct children of the root layout). If magazine-specific pages are added (e.g., `/magazine/[id]/posts`), create a separate intercepting route at that depth or accept full-page navigation there.

**Warning signs:**
- Clicking a card in a nested editorial context triggers full-page navigation instead of modal overlay
- No modal appears, URL changes to `/posts/[id]` with full page render
- `@modal` slot renders `null` (default.tsx returns null)

**Phase to address:**
Explore detail page connection phase — verify the interception boundary before adding any nested editorial routes.

### Pitfall 5: `spot_count` is `0` for all cards — N+1 solution data pattern

**What goes wrong:**
`useInfinitePosts` maps each Supabase post row to `PostGridItem` with `spotCount: 0` hardcoded (line 236 in `useImages.ts`). The REST API `PostListItem` model does include `spot_count` from the backend. However, solution/item detail per spot (brand, product name, thumbnail) is NOT included in the list endpoint — it requires a separate `GET /api/v1/posts/{post_id}` call per card. If each visible card makes its own detail fetch to show spot/solution data on hover, the grid will issue 40+ requests per page load — a classic N+1.

**Why it happens:**
List endpoints are intentionally lightweight. The backend `PostListItem` includes `spot_count` but not `SpotWithTopSolution[]` (which is only in `PostDetailResponse`). There is no batch endpoint.

**How to avoid:**
Use `spot_count` from the list response to show a count badge only (no individual solution details on cards). Load solution detail lazily — on hover or on modal open, not on grid render. Do not prefetch all visible cards' detail responses. The existing `usePostDetailForImage` hook is the correct call site for detail data; call it only when the modal is opened.

**Warning signs:**
- Network tab shows 40+ `/api/v1/posts/{id}` calls on grid render
- Grid initial load time exceeds 3s
- `ExploreCardCell` imports `usePostDetailForImage`

**Phase to address:**
Card enrichment phase (spot/solution display on cards) — define "card enrichment" as count-only from list data, not solution thumbnails.

### Pitfall 6: `hasMagazine` prop creates implicit coupling between Editorial and Explore behaviors

**What goes wrong:**
`ExploreClient` currently accepts a `hasMagazine?: boolean` prop that changes both the data filter and the editorial title overlay rendering (`item?.editorialTitle`). As editorial requirements grow (different sort orders, different empty states, different filter bars, different card layouts), the single `hasMagazine` flag will accumulate conditional branches. The component will become harder to test and reason about.

**Why it happens:**
Reusing `ExploreClient` with a prop is the correct choice for MVP (avoids duplication). But it tempts you to add more editorial-specific props over time: `hasMagazine`, `editorialSort`, `editorialLayout`, `showMagazineHeader`, etc.

**How to avoid:**
For v11.0, keep the single `hasMagazine` prop. Draw a firm line: the prop changes ONLY the data filter and the editorial title overlay. If editorial needs a different filter bar, different card component, or different header — create a thin `EditorialClient` wrapper that composes `ExploreClient` rather than adding more props to `ExploreClient`.

**Warning signs:**
- `ExploreClient` receives more than 3 editorial-specific props
- `hasMagazine && someOtherEditorialFlag` conditionals in JSX
- Editorial filter behavior diverges from Explore filter behavior

**Phase to address:**
Editorial data filtering phase — define the prop contract boundary as part of implementation.

### Pitfall 7: `usePostDetailForImage` uses Supabase direct query — `post_magazine_id` is never returned

**What goes wrong:**
`ImageDetailModal` reads `post_magazine_id` from the image data to drive the magazine-aware drawer content (magazine title, brands, style tags). `usePostDetailForImage` fetches via `fetchPostWithSpotsAndSolutions` which is a Supabase direct query on the `posts` table. The Supabase `PostRow` type does NOT include `post_magazine_id` in the mapped return type — meaning `magazineId` will always be `undefined`, and the magazine metadata section in the modal drawer will never render, regardless of how many posts have magazine associations.

**Why it happens:**
The Supabase query layer predates the magazine feature. The REST API `PostDetailResponse` does include `post_magazine_id`, but `usePostDetailForImage` doesn't use the REST endpoint.

**How to avoid:**
Migrate `usePostDetailForImage` to call `getPost` (the Orval-generated REST fetch function) instead of `fetchPostWithSpotsAndSolutions`. Map `PostDetailResponse` to the `ImageDetail` shape. This unblocks magazine metadata in the drawer and is the correct long-term direction anyway. The `postDetailToImageDetail` adapter already exists at `lib/api/adapters/postDetailToImageDetail.ts` — check if it handles the full mapping.

**Warning signs:**
- `magazineId` is always `undefined` in `ImageDetailModal` even for posts with `post_magazine_id`
- Magazine title section never renders in the drawer
- `magazine?.layout_json` is always null

**Phase to address:**
Explore detail page connection phase — before testing the full editorial drawer flow, confirm `post_magazine_id` is present in detail response.

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Keep Supabase direct query in `useInfinitePosts` with `// hasMagazine not supported` | Unblocks MVP | Two code paths for post fetching; Supabase query bypasses backend RLS at the API level | Only if REST API `has_magazine` param is genuinely undeliverable before deadline |
| Use Orval `useListPosts` hook directly without infinite composition | Simplest import path | No infinite scroll; requires full page reload to see more content | Never |
| Hardcode `spotCount: 0` on all cards | No N+1 requests | Spot count badge always "0"; misleads users about content richness | Only if spot count badge is not part of v11.0 scope |
| Skip `openapi.json` regeneration, manually add `has_magazine` to `ListPostsParams` | Faster than Rust rebuild | `listPostsParams.ts` diverges from spec; future codegen overwrites the manual addition | Never |
| Add `editorialSort` prop to `ExploreClient` instead of creating `EditorialClient` | One component to maintain | Accelerating prop accumulation; both modes become harder to test | Only for a single, non-repeating exception |
| Keep `usePostDetailForImage` on Supabase | No migration effort | Magazine metadata never works in drawer; editorial content unfinished | Never if magazine drawer content is in scope |

| `postDetailToImageDetail` adapter | Write a new adapter from scratch | Check `lib/api/adapters/postDetailToImageDetail.ts` — adapter may already exist for REST → ImageDetail mapping |
