## Phase Details

### Phase 44: Memory Leak Prevention

**Goal**: 사용자가 앱을 장시간 사용하거나 모달을 빠르게 열고 닫아도 메모리 누수, 애니메이션 충돌, 잘못된 상태 전환이 발생하지 않는다
**Depends on**: Nothing (first phase of v10.0)
**Requirements**: MEM-01, MEM-02, MEM-03, MEM-04, MEM-05
**Success Criteria** (what must be TRUE):
1. 이미지 업로드 중 사용자가 페이지를 이탈하면 `URL.revokeObjectURL`이 자동으로 호출되어 메모리가 해제된다 (`beforeunload` + `useEffect` cleanup 확인 가능)
2. 이미지 디테일 모달을 10회 빠르게 열고 닫아도 GSAP 애니메이션 충돌 없이 정상 작동한다 (`useGSAP` 훅의 `revert()` 자동 보장)
3. requestStore의 `setStep()`에 잘못된 순서를 전달하면 (예: step 1 → step 3 건너뜀) 상태 변경이 거부되고 에러가 발생한다
4. `setDetectedSpots()` 호출 후 원본 배열을 외부에서 수정해도 store의 spots 데이터가 변경되지 않는다 (불변성 보장)
5. 데스크탑 브라우저에서 이미지 디테일 모달의 이미지 영역을 스크롤하면 오른쪽 콘텐츠 패널이 함께 스크롤된다

### Phase 45: Security Foundation

**Goal**: `POST /api/v1/posts/analyze` 엔드포인트가 남용으로부터 보호되고, 디버그 정보가 프로덕션 브라우저 콘솔에 노출되지 않으며, 환경 변수 누락 시 배포가 조용히 실패하지 않는다
**Depends on**: Phase 44 (GSAP cleanup needed before refactoring; env validation reduces noise in observability phase)
**Requirements**: SEC-01, SEC-02, SEC-03, SEC-04
**Success Criteria** (what must be TRUE):
1. 동일 IP에서 `POST /api/v1/posts/analyze`를 60초 안에 11회 이상 호출하면 11번째 요청부터 429 응답이 반환된다
2. 프로덕션 빌드(`NODE_ENV=production`)에서 `ImageDetailModal`의 `imageId`, `image`, `error` 콘솔 출력이 사라진다
3. `API_BASE_URL` 환경 변수가 없는 상태로 Next.js 서버를 시작하면 앱 로드 전 명확한 에러 메시지와 함께 프로세스가 종료된다
4. 백엔드가 422 응답을 반환하는 요청을 보냈을 때 브라우저는 500이 아닌 422 상태 코드와 원본 에러 body를 받는다

### Phase 46: Component Refactoring

**Goal**: `ThiingsGrid`의 물리 엔진이 React와 분리되어 독립 테스트가 가능하고, `ImageDetailModal`의 애니메이션 로직이 커스텀 훅으로 분리되어 메인 파일이 200줄 이하가 되며, `DecodedLogo`가 WebGL 미지원 환경에서도 정상 렌더링된다
**Depends on**: Phase 44 (GSAP cleanup in MEM-02 must be complete before extracting animation hook in REF-02)
**Requirements**: REF-01, REF-02, REF-03
**Success Criteria** (what must be TRUE):
1. `lib/utils/thiingsGridPhysics.ts`가 존재하며 React를 import하지 않는다; 순수 TypeScript로 velocity, friction, grid cell 계산 함수를 export한다
2. `ImageDetailModal.tsx`의 줄 수가 200줄 이하이며, GSAP 타임라인 코드가 `useDrawerAnimation` 훅 파일에 위치한다
3. WebGL이 비활성화된 브라우저(인스타그램 인앱 브라우저 시뮬레이션)에서 `/` 페이지를 열면 로고가 SVG 또는 정적 이미지로 정상 표시된다
4. 리팩터링 전/후로 `bun run build`와 `bun run typecheck`가 모두 오류 없이 통과한다

### Phase 47: Observability

**Goal**: 프로덕션에서 발생하는 프론트엔드/백엔드 에러가 Sentry 대시보드에 집계되고, Core Web Vitals 지표가 수집되어 성능 회귀를 감지할 수 있다
**Depends on**: Phase 45 (debug logging removed — SEC-02, proxy error forwarding in place — SEC-04; without these, Sentry sees noise not signal)
**Requirements**: OBS-01, OBS-02, OBS-03, OBS-04
**Success Criteria** (what must be TRUE):
1. 프론트엔드에서 의도적으로 throw한 에러가 1분 이내에 Sentry 프로젝트 Issues 탭에 나타나며, 브라우저 콘솔에는 토큰이나 이메일이 포함되지 않는다
2. Rust API에서 `tracing::error!`로 기록된 이벤트가 Sentry에서 프론트엔드 에러와 연관 trace로 조회된다
3. 메인 페이지 방문 시 LCP, CLS 지표가 수집되어 Sentry Performance 또는 지정 엔드포인트에 전송된다
4. 프록시 라우트가 백엔드의 422 에러를 반환할 때 Sentry에서 해당 에러의 실제 HTTP 상태 코드(422)와 에러 body가 포함된 이벤트가 기록된다
**Plans**: TBD

### Phase 48: Test Coverage

**Goal**: requestStore 상태 전환이 자동화된 테스트로 검증되고, 인증/업로드/모달 핵심 경로가 Playwright E2E 테스트로 커버되어 회귀를 자동으로 감지한다
**Depends on**: Phase 44 (MEM-01, MEM-02, MEM-03 — tests verify the fixes work), Phase 46 (REF-01 — thiingsGridPhysics extractable to unit test)
**Requirements**: REF-04, TEST-01, TEST-02, TEST-03, TEST-04
**Success Criteria** (what must be TRUE):
1. `packages/web/__tests__/stores/requestStore.test.ts`가 존재하며 step 순서 위반 케이스에서 테스트가 실패(→ 구현이 reject함을 확인)한다
2. `apiToStoreCoord(0, 0)`, `apiToStoreCoord(100, 100)`, 소수점 입력에 대한 단위 테스트가 통과하며, `thiingsGridPhysics` 클래스의 `applyFriction()` 함수 테스트가 통과한다
3. Playwright E2E 테스트가 Google OAuth 리다이렉트 완료 후 세션 쿠키 존재를 확인하고, 모달 open → close → escape 키 사이클을 에러 없이 통과한다
4. Playwright E2E 테스트가 이미지 업로드 → AI 분석 요청 → 분석 결과 표시 전체 흐름을 커버하며, `bun run test:e2e` 명령으로 CI에서 실행 가능하다
5. `bun run size` 명령이 설정된 번들 사이즈 임계값을 측정하여 초과 시 non-zero exit code를 반환한다
