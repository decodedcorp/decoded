  <files>specs/_shared/injection-guide.md</files>
  <action>
Create `specs/_shared/injection-guide.md` — the definitive protocol for AI agents to determine which spec files to load for any given task.

**Context:** The README already has a quick-reference injection table (from v4-01-02). This guide is the "full guide" that README links to. After creating this file, the README's injection section should reference this file as the single source of truth.

**Steps:**

1. Design task type classifications based on actual codebase work patterns:
   - **Modify existing screen** — changing UI, layout, behavior of an existing page
   - **Add new feature to screen** — extending a page with new functionality
   - **Fix bug** — debugging and resolving issues
   - **Create new screen** — building a page from scratch
   - **Modify data flow** — changing API calls, store state, data transformations
   - **Update shared component** — modifying a design system component
   - **Understand architecture** — learning how the system works

2. For each task type, specify:
   - **Required files** (must load) — with exact spec file paths
   - **Recommended files** (load if relevant) — with conditions
   - **Loading order** — which files to read first for maximum context efficiency

3. Structure as a decision tree + reference tables:

   ```
   What are you doing?
   ├── Modifying a screen → Load screen spec + component-registry + store-map
   ├── Adding a feature → Load flow spec + screen spec + api-contracts
   ├── Fixing a bug → Load screen spec + component-registry + store-map
   ├── Creating new screen → Load template + component-registry + relevant flow
   ├── Modifying data flow → Load api-contracts + data-models + store-map
   ├── Updating component → Load component-registry + affected screen specs
   └── Understanding architecture → Load store-map + api-contracts + data-models
   ```

4. Include a **File Inventory** table — all spec files with their purpose:

   | File | Path | Purpose |
   |------|------|---------|
   | Component Registry | `specs/_shared/component-registry.md` | Design system lookup |
   | Data Models | `specs/_shared/data-models.md` | Entity type reference |
   | API Contracts | `specs/_shared/api-contracts.md` | Endpoint reference |
   | Store Map | `specs/_shared/store-map.md` | State management reference |
   | Screen Spec Template | `specs/shared/templates/screen-spec-template.md` | Template for new specs |
   | Flow Spec Template | `specs/shared/templates/flow-spec-template.md` | Template for new flows |

5. Include a **Loading Priority** section:
   - Always load the most specific file first (screen spec before shared files)
   - Shared files provide context but screen specs define requirements
   - For cross-screen work, load the flow document first

6. Include a **Context Budget** note:
   - Each shared file is designed to be under 400 lines
   - Loading all 5 shared files ≈ ~1,500 lines of context
   - Screen specs target 200 lines each
   - Recommend loading 1 screen spec + 2-3 shared files per task (~800 lines)

**IMPORTANT:**
- This file replaces the quick-reference table in README as the single source of truth
- After creating this file, do NOT modify README — the link from README to this file is already correct
- File paths in examples must use actual spec file paths (specs/_shared/*, specs/screens/*, specs/flows/*)
