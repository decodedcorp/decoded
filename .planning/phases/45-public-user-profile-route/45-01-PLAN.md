Create `packages/web/app/profile/[userId]/PublicProfileClient.tsx` as a Client Component. This is a standalone component (NOT extending ProfileClient) to avoid Zustand profileStore pollution.

**CRITICAL ARCHITECTURE DECISION:** Do NOT use `ProfileHeader`, `StatsCards`, `BadgeGrid`, `RankingList`, or `ArchiveStats` components directly — they all read from `useProfileStore` (the logged-in user's Zustand store) and would show the WRONG user's data. Instead:

1. Use `ProfileHeaderCard` (design-system primitive) directly with data from `useUser(userId)` — this accepts explicit props, no store dependency.
2. For stats display, render a simple inline 3-column grid with data from `UserResponse` (total_points, rank).
3. For `BadgeGrid` and `RankingList`: skip rendering — no API endpoint exists for other users' badges/rankings. Show nothing (not an empty state, just omit).
4. For `ArchiveStats`: skip rendering — it reads from profileStore (wrong user data). The UserResponse does not contain post/spot counts.

**Components to render (all accept explicit props):**
- `ProfileHeaderCard` from `@/lib/design-system` — with user data from `useUser(userId)`, NO actions slot (no edit/logout buttons)
- `ProfileBio` — with `bio={userData?.bio ?? undefined}`
- `FollowStats` — with `followers={0} following={0}` (real data in Phase 46+)
