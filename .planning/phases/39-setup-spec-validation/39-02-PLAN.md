    **Step 2: SPEC-01 — OpenAPI version and nullable audit**

    Run these exact commands and capture output:

    ```bash
    # 1. Check spec version
    jq '.openapi' packages/api-server/openapi.json

    # 2. Count oneOf patterns (OpenAPI 3.1 nullable style — expected > 0)
    jq '[.. | objects | select(.oneOf != null)] | length' packages/api-server/openapi.json

    # 3. Count nullable:true patterns (OpenAPI 3.0 style — expected 0)
    jq '[.. | objects | select(.nullable == true)] | length' packages/api-server/openapi.json
    ```

    Interpretation:
    - If version is "3.1.0" AND oneOf count > 0 AND nullable count = 0: utoipa 5.x uses 3.1 canonical nullable. Orval 8.5.3 handles this natively. No preprocessing needed.
    - If version is "3.0.x": Different handling may be needed (unlikely based on research).
    - If nullable count > 0: Mixed patterns — may need transformer.

    **Step 3: SPEC-02 — operationId uniqueness audit**

    ```bash
    # List all operationIds
    jq '[.paths | to_entries[] | .value | to_entries[] | .value.operationId] | sort' packages/api-server/openapi.json

    # Find duplicates (should output empty array [])
    jq '[.paths | to_entries[] | .value | to_entries[] | .value.operationId] | sort | group_by(.) | map(select(length > 1)) | flatten' packages/api-server/openapi.json

    # Count total endpoints
    jq '[.paths | to_entries[] | .value | keys[] | select(. != "parameters")] | length' packages/api-server/openapi.json

    # List all tags
    jq '[.tags[].name] | sort' packages/api-server/openapi.json
    ```

    If duplicates are found:
    - Document each duplicate (which paths share the operationId)
    - Record as a BLOCKER for Phase 40 — backend needs explicit `operation_id = "..."` attributes added to the utoipa `#[utoipa::path]` annotations on the duplicate handlers
    - Specifically check for: `list_posts` (posts vs admin::posts), `list_badges` (badges vs admin::badges)

    **Step 4: SPEC-05 — Pagination parameter audit**

    ```bash
    # Find all endpoints with page/per_page parameters
    jq '.paths | to_entries[] | {path: .key, methods: [.value | to_entries[] | select(.key != "parameters") | select(.value.parameters != null) | {method: .key, params: [.value.parameters[]? | select(.name == "page" or .name == "per_page") | .name]}] | select(length > 0)}' packages/api-server/openapi.json

    # Check for any cursor-based pagination parameters
    jq '[.paths | to_entries[] | .value | to_entries[] | select(.value.parameters != null) | .value.parameters[]? | select(.name == "cursor" or .name == "after" or .name == "before") | .name] | unique' packages/api-server/openapi.json
    ```

    Expected: All paginated endpoints use `page` + `per_page`. No cursor/after/before params.

    **Step 5: Verify multipart endpoints match exclusion list**

    ```bash
    # Find all paths with multipart/form-data request bodies
    jq '[.paths | to_entries[] | {path: .key, methods: [.value | to_entries[] | select(.value.requestBody?.content["multipart/form-data"] != null) | {method: .key, operationId: .value.operationId}]} | select(.methods | length > 0)]' packages/api-server/openapi.json
    ```

    Verify the output matches exactly the 4 multipart endpoints in the orval.config.ts transformer:
    - POST /api/v1/posts (create_post_without_solutions)
    - POST /api/v1/posts/with-solutions (create_post_with_solutions)
    - POST /api/v1/posts/upload (upload_image)
    - POST /api/v1/posts/analyze (analyze_image)

    If there are additional multipart endpoints not in the exclusion list, update the audit document with the new paths that need to be added to orval.config.ts.
