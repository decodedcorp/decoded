Create `specs/shared/templates/screen-spec-template.md` -- the new v4.0 screen spec template.

Design principles (from STATE.md decisions):
- **EARS syntax**: Requirements use "When [trigger], the system shall [behavior]" format
- **Mobile-first**: Mobile wireframe comes first, desktop is the responsive adaptation
- **200-line target** (300 max with justification): Compact, no padding
- **AI context optimization**: Structured for injection into AI agent prompts
- **Component mapping with verified file paths**: Every component references actual `packages/web/` paths
- **Anti-features to EXCLUDE**: i18n sections, version history tables, copied TypeScript types, placeholder sections, test scenario tables, implementation checklists

Template structure (target ~60 lines of template):

```markdown
# [SCR-XXX-NN] Screen Name

> Route: `/path` | Status: implemented | Updated: YYYY-MM-DD

## Purpose

One sentence describing what the user accomplishes on this screen.

## Component Map

| Region | Component | File | Props/Notes |
|--------|-----------|------|-------------|
| Header | MobileHeader | `lib/design-system/mobile-header.tsx` | variant="default" |
| Hero | HeroSection | `lib/components/main/hero-section.tsx` | posts={Post[]} |
| ... | ... | ... | ... |

## Layout

### Mobile (default)

[ASCII wireframe - compact, showing component names from Component Map]

### Desktop (≥768px)

[Brief description of layout changes OR compact wireframe if significantly different]

## Requirements

### Data Loading
- When the screen mounts, the system shall fetch [endpoint] and display [data].
- When the fetch fails, the system shall show [error state].

### User Interactions
- When the user [action], the system shall [behavior].
- When [condition], the system shall [behavior].

### State
- Store: `[storeName]` from `lib/stores/[store].ts`
- Key fields: `field1`, `field2`
- Query key: `["queryKey", param]`

## Navigation

| Trigger | Destination | Data Passed |
|---------|-------------|-------------|
| Card click | `/posts/[id]` | postId |
| Search icon | `/search` | - |

## Error & Empty States

| State | Condition | UI |
|-------|-----------|-----|
| Loading | Initial fetch | Skeleton ([SkeletonCard]) |
| Empty | No results | "No items found" message |
| Error | API 4xx/5xx | Error message + retry |

## Animations (if applicable)

| Trigger | Type | Library | Notes |
|---------|------|---------|-------|
| Scroll | Parallax | GSAP ScrollTrigger | Hero section only |
```

Key differences from old template:
- NO i18n section, NO version history, NO implementation checklist, NO test scenarios
- NO copied TypeScript type definitions
- EARS-style requirements instead of free-form descriptions
- Component Map with VERIFIED file paths (executor must verify paths before writing)
- Mobile-first layout (mobile wireframe is primary)
- Compact -- template itself should be ~60 lines, real specs target 150-200 lines

**Part A: Create flow spec template**

Create `specs/shared/templates/flow-spec-template.md` for FLW-* flow documents.

Flow spec template structure (~50 lines):

```markdown
# [FLW-NN] Flow Name

> Screens: SCR-XXX-01 → SCR-XXX-02 → SCR-XXX-03 | Updated: YYYY-MM-DD

## Journey

Brief description of the user journey this flow covers.

## Screen Sequence

```
[Screen A] --action--> [Screen B] --action--> [Screen C]
     |                      |
     v                      v
  [Alt path]           [Error state]
```

## Steps

### Step 1: [Action Name]

- **Screen:** SCR-XXX-01 (`/route`)
- **Trigger:** User does [action]
- **State change:** `store.field` changes from A to B
- **Data:** POST `/api/v1/endpoint` with { payload }
- **Next:** → Step 2 (on success) | → Error (on failure)

### Step 2: [Action Name]
...

## State Transitions

| From | Event | To | Side Effect |
|------|-------|----|-------------|
| idle | user clicks upload | uploading | POST /api/v1/upload |
| uploading | success response | detecting | trigger AI analysis |
| detecting | analysis complete | review | populate spots[] |

## Shared Data

| Data | Source | Consumed By |
|------|--------|-------------|
| postId | Step 1 response | Step 2, Step 3 |
| spots[] | Step 2 AI result | Step 3 review UI |

## Error Recovery

| Error Point | Recovery | Fallback |
|-------------|----------|----------|
| Upload fails | Retry with toast | Return to Step 1 |
```
