**ItemDetailCard.tsx (line 95) -- setTimeout(0) for click event registration:**
- Current: `const tid = setTimeout(() => { document.addEventListener("click", handleClickOutside); }, 0);`
- Replace with direct registration: `document.addEventListener("click", handleClickOutside);`
- Remove `clearTimeout(tid)` from cleanup, keep `document.removeEventListener("click", handleClickOutside)`
- Add `handleClickOutside` to useEffect dependency array if not present
- Reason: click events already fire after the current event loop tick; setTimeout(0) delay is unnecessary

**TrendingListSection.tsx (line 80) -- setTimeout(50) for GSAP init:**
- Current: `const timer = setTimeout(setup, 50);` with `clearTimeout(timer)` in cleanup
- Replace with: `const rafId = requestAnimationFrame(setup);` with `cancelAnimationFrame(rafId)` in cleanup
- Reason: requestAnimationFrame ensures layout is computed before GSAP reads DOM measurements

**useImageUpload.ts (line 84) -- setTimeout(100) for startDetection:**
- Current: `setTimeout(() => { startDetection(); }, 100);`
- Replace with direct call: `startDetection();`
- Reason: Zustand state updates are synchronous; no need to wait for UI update

**ImageDetailModal.tsx (line 129) -- setTimeout(50) for transition:**

- Replace with GSAP `onComplete` callback or `gsap.delayedCall(0.05, () => { ... })` which is properly cleaned up by the GSAP context
- Reason: gsap.delayedCall is context-managed and auto-cleaned on revert()

**AISummarySection.tsx (line 14) -- setTimeout(300) for reveal:**
- Current: `const t = setTimeout(() => setRevealed(true), 300); return () => clearTimeout(t);`
- This has proper cleanup and is a simple delay pattern, not a workaround

- Reason: already has cleanup, not a GSAP context issue, and the delay is intentional for animation sequencing

**StudioLoader.tsx (line 21) -- setTimeout(300) for unmount delay:**

**VtonModal.tsx (line 344) -- setTimeout(300) debounce:**
