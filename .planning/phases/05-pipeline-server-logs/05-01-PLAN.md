**pipeline-mock-data.ts:**

Types:
- `PipelineStatus`: `'completed' | 'running' | 'failed'`
- `StepStatus`: `'completed' | 'running' | 'failed' | 'pending'`
- `PipelineStep`: `{ name: string; status: StepStatus; startedAt: string; completedAt?: string; durationMs?: number; error?: string; result?: Record<string, unknown> }`
  - Step names: `'upload'`, `'analyze'`, `'detect'`
- `PipelineExecution`: `{ id: string; postId: string; imageUrl: string; status: PipelineStatus; startedAt: string; completedAt?: string; totalDurationMs?: number; steps: PipelineStep[]; triggerUser: string; }`
- `PipelineListItem`: `Omit<PipelineExecution, 'steps'>` (list endpoint omits step detail for performance, same as audit pattern)

Generator:
- `generatePipelines()`: Creates 30 deterministic pipeline executions using djb2 hash seeding
- Implement `deterministicInt(seed, min, max)` locally (same as ai-cost-mock-data.ts pattern — NOT a shared import)
- Status distribution: ~22 completed, ~3 running, ~5 failed (realistic proportions)
- Each pipeline has 3 steps in order: upload → analyze → detect
  - Completed pipelines: all 3 steps completed with realistic durations (upload: 1-5s, analyze: 3-15s, detect: 2-8s)
  - Running pipelines: first 1-2 steps completed, current step status 'running', remaining steps 'pending'
  - Failed pipelines: steps completed up to failure point, failed step has error message and status 'failed', remaining steps 'pending'
- `imageUrl`: Use `https://picsum.photos/seed/{hash}/400/300` for deterministic placeholder images
- `triggerUser`: Deterministic user names from a fixed array of 8 Korean names
- Use module-level cache singleton (`_cachedPipelines`) like audit-mock-data.ts for stability
- `getPipelineById(id)`: Lookup from cached data, return full PipelineExecution with steps
- Error messages for failed steps: realistic strings like "Vision API timeout after 30s", "Image format not supported: .webp", "Rate limit exceeded (429)", "Model inference OOM"

**pipeline.ts:**
- Import types from pipeline-mock-data.ts
- Re-export types for UI consumers (single import source)
- `PipelineListResponse`: `{ data: PipelineListItem[]; total: number; page: number; pageSize: number; }`
- `fetchPipelines(params)`: Filter by status, paginate (page/pageSize defaults 1/15), return PipelineListResponse
- `fetchPipelineDetail(id)`: Return full PipelineExecution with steps or null

**API route /api/v1/admin/pipeline/route.ts (GET):**
- Admin auth check using checkIsAdmin pattern from existing routes
- Query params: `?status=completed|running|failed&page=1&pageSize=15`
- Returns JSON `PipelineListResponse`

**API route /api/v1/admin/pipeline/[pipelineId]/route.ts (GET):**
- Admin auth check
- Returns full PipelineExecution with steps
- 404 if not found

Create server log mock data and API infrastructure.

**server-logs-mock-data.ts:**

Types:
- `LogLevel`: `'info' | 'warn' | 'error' | 'debug'`
- `ServerLogEntry`: `{ id: string; timestamp: string; level: LogLevel; message: string; method: string; endpoint: string; statusCode: number; responseTimeMs: number; userId?: string; userAgent?: string; ip?: string; }`

Generator:
- `generateServerLogs(count?: number)`: Creates 200 deterministic log entries using djb2 hash seeding
- Implement `deterministicInt(seed, min, max)` locally
- Level distribution: ~60% info, ~20% warn, ~12% error, ~8% debug
- Realistic endpoint pool: `['/api/v1/posts', '/api/v1/posts/{id}', '/api/v1/users/me', '/api/v1/categories', '/api/v1/posts/analyze', '/api/v1/posts/upload', '/api/v1/solutions/{id}', '/api/v1/spots/{id}', '/api/v1/search']`
- HTTP methods: GET (60%), POST (25%), PATCH (10%), DELETE (5%)
- Status codes: 200/201 for info, 400/401/403 for warn, 500/502/503 for error
- Response times: info 50-300ms, warn 200-800ms, error 500-5000ms
- Messages: Realistic log messages matching the endpoint and level, e.g.:
  - info: "GET /api/v1/posts 200 145ms"
  - warn: "Slow query: GET /api/v1/posts/analyze 823ms"
  - error: "Internal server error: database connection timeout"
- Timestamps: Spread over last 24 hours with more recent entries having higher density
- Use module-level cache singleton
- `generateStreamLogs(sinceId?: string)`: Return 1-3 NEW log entries with timestamps after the most recent entry or after sinceId. These should NOT be deterministic — use `Date.now()` for realistic streaming feel. Each call generates fresh entries.

**server-logs.ts:**
- Import types from server-logs-mock-data.ts
- Re-export types for UI consumers
- `ServerLogListResponse`: `{ data: ServerLogEntry[]; total: number; page: number; pageSize: number; }`
- `fetchServerLogs(params)`: Filter by level, search text (message/endpoint match), date range (from/to ISO strings), paginate (page/pageSize defaults 1/50)
- `fetchLogStream(sinceId?)`: Return array of new log entries for polling

**API route /api/v1/admin/server-logs/route.ts (GET):**
- Admin auth check
- Query params: `?level=info|warn|error|debug&search=text&from=ISO8601&to=ISO8601&page=1&pageSize=50`
  - `from`/`to` are optional ISO 8601 date strings for date range filtering. When provided, only return log entries whose timestamp falls within the range.
- Returns JSON `ServerLogListResponse`

**API route /api/v1/admin/server-logs/stream/route.ts (GET):**
- Admin auth check
- Query params: `?since_id=xxx` (optional, returns entries after this ID)
- Returns JSON `{ entries: ServerLogEntry[]; latestId: string }`
- This endpoint is called every 2-3 seconds by the streaming UI
